# 컨텍스트 갱신: CLAUDE.md 증분 업데이트

CLAUDE.md를 처음부터 다시 만들지 않고, **마지막 문서 갱신 이후 실제로 바뀐
부분만** 찾아서 수정한다.

## 절차

1. **변경 범위 파악**: CLAUDE.md의 마지막 커밋 이후 무엇이 바뀌었는지 확인한다.

   ```bash
   LAST=$(git log -1 --format=%H -- CLAUDE.md)
   git diff --stat "$LAST"..HEAD          # 어떤 파일이 얼마나 바뀌었나
   git log --oneline "$LAST"..HEAD        # 어떤 커밋들이 있었나
   ```

2. **문서 영향 분석**: 변경사항을 CLAUDE.md의 각 섹션과 대조한다.
   - 매니페스트/CI 설정이 바뀌었나 → **명령어** 섹션 검증 (바뀐 명령은 실행해서 확인)
   - 디렉터리가 추가/삭제/이동되었나 → **아키텍처** 섹션 갱신
   - 새 패턴이 도입되었나 (커밋 메시지에 refactor, migrate 등) → **컨벤션** 섹션 검토
   - 새 환경변수·설정 파일이 생겼나 → **주의사항** 섹션 갱신

3. **최소 수정**: 영향받은 섹션만 고친다. 문서 전체를 다시 쓰지 마라.
   사용자가 직접 추가한 커스텀 지침은 절대 삭제하지 않는다.

4. **보고**: 무엇을 왜 바꿨는지 diff 요약과 함께 보고한다. 바뀐 게 없으면
   "문서가 최신 상태"라고만 답하고 파일을 건드리지 않는다.

## 자동 호출 모드 (훅에 의해 조용히 실행되는 경우)

세션 스냅샷 훅의 지시로 사용자 요청 완료 후 자동 실행되는 경우:

- **묻지 않는다**: 실행 여부·수정 내용에 대해 사용자 확인을 구하지 않는다.
- **방금 작업의 지식을 재활용한다**: 이미 읽은 파일을 다시 읽지 않는다.
- **커밋하지 않는다**: 작업 트리에만 남겨 사용자가 diff로 검토할 수 있게 한다.
- **보고는 한 줄**: 최종 보고 끝에 "CLAUDE.md를 최신화했습니다" 정도만 덧붙인다.
  바뀐 게 없으면 아무 말도 하지 않는다.

## 판단 기준

- 변경 커밋이 매우 많거나(예: 100+), 프로젝트 구조가 근본적으로 바뀌었으면
  증분 갱신 대신 `/project-onboard`로 전체 재생성을 제안한다.
- CLAUDE.md가 아예 없으면 이 스킬이 아니라 `/project-onboard`를 실행한다.
