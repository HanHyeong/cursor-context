# 프로젝트 온보딩: 자동 컨텍스트 문서 생성

프로젝트 전체를 분석해서 `.cursor-context/project-context.md`를 생성한다. 이 파일은
세션 시작 훅이 매 세션 컨텍스트에 자동 주입하는 **기계 생성 산출물**이다.
목표는 **"이 문서만 읽으면 신입 개발자가 바로 작업을 시작할 수 있는 수준"**의 문서다.

## 출력 언어

`.cursor-context/config`에 `LANG=en`이 설정돼 있으면(기본값은 en이다 —
README가 영어 우선이므로) project-context.md **본문**을 한국어 대신
영어로 작성한다. 이 SKILL.md 자체의 절차 설명은 번역 대상이 아니다 —
읽는 주체가 사람이 아니라 Claude이므로 언어와 무관하게 동일하게 이해한다.
config가 없거나 `LANG=ko`면 한국어로 작성한다.

## 소유권 규칙 (가장 중요)

- **출력 대상은 오직 `.cursor-context/project-context.md`이다.**
- **CLAUDE.md는 사용자가 관리하는 지침 파일이다. 읽는 것은 좋지만, 사용자가
  명시적으로 요청하지 않는 한 절대 수정·생성·삭제하지 않는다.**
- CLAUDE.md에 이미 적힌 내용(사용자 지침)과 중복되는 내용은 project-context.md에
  다시 쓰지 않는다. 두 파일이 충돌하면 CLAUDE.md(사용자 지침)가 우선이다.

## 1단계: 병렬 탐색

다음 항목을 조사한다. 프로젝트가 크면 Explore 서브에이전트를 병렬로 띄워서 조사하라:

1. **기술 스택과 빌드 시스템**: 매니페스트 파일(package.json, pyproject.toml, go.mod 등),
   락파일, 빌드 설정, CI 워크플로우(.github/workflows/)를 읽고 다음을 확정:
   - 빌드/실행/테스트/린트 명령어 (실제로 CI에서 쓰는 명령어 우선)
   - 언어 버전, 패키지 매니저
2. **아키텍처**: 진입점(main, index, app 등)부터 따라가며 주요 디렉터리의 역할,
   레이어 구분(라우팅/서비스/데이터 등), 모듈 간 의존 방향을 파악
3. **코드 컨벤션**: 대표 소스 파일 5~10개를 읽고 다음을 파악:
   - 네이밍 규칙, 파일 배치 규칙, import 스타일
   - 에러 처리 패턴, 테스트 작성 패턴
   - 린터/포매터 설정 파일(.eslintrc, ruff.toml, .editorconfig 등)의 실제 규칙
4. **도메인 지식**: README, docs/, 주석에서 프로젝트가 해결하는 문제와 핵심 용어 파악
5. **함정(gotchas)**: 환경변수 요구사항, 마이그레이션 절차, 순서 의존적 명령,
   "왜 이렇게 했는지" 주석이 달린 특이한 코드

## 2단계: project-context.md 작성

**파일 상단에 반드시 신선도 마커를 넣는다** (훅이 이 마커로 문서가 오래됐는지
실시간 판단한다). 지문 블록은 반드시 지문 생성기 출력을 그대로 사용한다:

```bash
HEAD=$(git rev-parse HEAD)
FP=$(<hooks>/context-fingerprint.sh)
```

`<hooks>/`는 배치에 따라 해석한다: `.claude/hooks/`에 스크립트가 있으면
그것을(install.sh 배치), 없으면 이 SKILL.md 기준 `../../hooks/`를(플러그인
배치 — 두 배치 모두 skills/와 hooks/가 같은 루트의 형제라 이 상대 경로는
항상 성립한다) 쓴다.

```markdown
<!-- generated-at-commit: (위 HEAD 값) -->
<!-- context-fingerprint-begin
(위 FP 출력을 그대로 붙여넣기 — 구조적 파일들의 내용 해시 목록)
context-fingerprint-end -->
<!-- 이 파일은 project-onboard 스킬이 자동 생성했습니다. 직접 편집해도 되지만 다음 자동 갱신 때 덮어써질 수 있습니다. 영구 지침은 CLAUDE.md에 적으세요. -->
```

지문 블록은 훅이 매 세션 작업 트리의 실제 내용과 비교하는 기준이므로,
직접 계산하거나 수정하지 말고 반드시 생성기 출력을 사용한다.
생성기 출력이 비어 있으면(종료 코드 3 — 해시 도구 없는 환경) 지문 블록을
아예 넣지 마라 — 빈 블록은 거짓 불일치를 만든다. generated-at-commit
마커만 기록하면 된다.

이어서 아래 템플릿 구조로 작성하되, **해당 없는 섹션은 삭제**하고 실제 조사로
확인된 내용만 적는다. 추측한 내용을 사실처럼 쓰지 마라.

```markdown
# 프로젝트명 — 자동 생성 컨텍스트

한두 문장으로 이 프로젝트가 무엇인지.

## 명령어

| 작업 | 명령어 |
|------|--------|
| 개발 서버 | ... |
| 테스트 (전체) | ... |
| 테스트 (단일 파일) | ... |
| 린트/포맷 | ... |
| 빌드 | ... |

## 아키텍처

- `src/...` — 역할
- `src/...` — 역할
(핵심 데이터 흐름을 2~3문장으로: 요청이 들어와서 나가기까지 어떤 레이어를 거치는지)

## 컨벤션

- (린터가 잡아주지 못하는 규칙만: 파일 배치, 네이밍, 패턴 선택 기준)

## 주의사항

- (환경변수, 마이그레이션, 순서 의존 명령, 건드리면 안 되는 것)
```

## 작성 원칙

- **간결하게**: 이 문서는 매 세션 컨텍스트에 주입된다. 200줄 이내를 목표로,
  긴 내용은 `docs/` 파일로 빼고 경로만 언급한다 (훅은 250줄까지만 주입한다).
- **명령어는 실행 검증**: 테스트/린트 명령은 가능하면 실제로 실행해서 동작을 확인한 뒤 적는다.
  단, 실행 검증은 **부작용 없는 명령에 한정**한다 — 테스트·린트·타입체크·
  빌드(로컬 산출물만 만드는 것)까지만. 배포, publish, 마이그레이션, 데이터
  삭제·변경, 외부 서비스 호출처럼 상태를 바꾸는 명령은 절대 실행하지 않고
  존재 여부(스크립트 정의, CI에서의 사용)만 확인해서 적는다.
- **린터가 잡는 규칙은 쓰지 않는다**: 세미콜론, 들여쓰기 같은 것은 도구가 처리한다.
- **로컬 산출물로 취급**: 커서의 인덱스처럼 커밋하지 않는 것이 기본이다.
  `.gitignore`에 다음 두 줄을 추가한다 — 파일이 없으면 이 내용만 담아
  새로 생성한다 (추가·생성 모두 안전한 변경이므로 자동 모드에서도 허용;
  이걸 빼먹으면 매 세션 git status에 미추적 소음이 남는다). 팀과 공유하고
  싶다는 사용자 요청이 있을 때만 커밋한다:

  ```
  # cursor-context 로컬 산출물
  .cursor-context/
  ```

## 3단계: 보고

생성 후 사용자에게: 문서 위치, 파악한 핵심 내용 요약, 확신이 낮아 확인이 필요한 부분을 보고한다.

## 자동 호출 모드 (훅에 의해 조용히 실행되는 경우)

세션 스냅샷 훅의 지시로 사용자 요청 완료 후 자동 실행되는 경우, 위 절차에
다음 규칙을 덧붙인다 — 커서 IDE의 백그라운드 인덱싱처럼 동작하는 것이 목표다:

- **묻지 않는다**: 실행 여부·내용에 대해 사용자 확인을 구하지 않는다.
- **탐색을 아낀다**: 방금 작업에서 이미 읽은 파일·파악한 구조를 재활용하고,
  부족한 부분만 최소한으로 추가 탐색한다. 서브에이전트 병렬 탐색은 생략해도 된다.
- **커밋하지 않는다**: 생성 파일은 작업 트리에만 남긴다.
- **보고는 한 줄**: 최종 보고 끝에 "프로젝트 컨텍스트 문서를 생성했습니다
  (.cursor-context/project-context.md)" 정도만 덧붙인다. 문서 내용을 장황하게 설명하지 않는다.
