# 컨텍스트 갱신: project-context.md 증분 업데이트

`.cursor-context/project-context.md`를 처음부터 다시 만들지 않고, **마지막 생성 시점
이후 실제로 바뀐 부분만** 찾아서 수정한다. 커서의 증분 인덱싱처럼
**갱신은 작고 자주**가 원칙이다 — 밀린 변경을 모아서 크게 갱신하는 것보다
바뀐 즉시 조금씩 갱신하는 쪽이 싸고 정확하다.

## 세 가지 갱신 모드 (훅이 어떤 지시를 줬는지에 따라)

1. **미세 갱신**: 작업 중 문서와 실제 코드의 불일치를 발견한 경우.
   diff 분석 없이 발견한 그 부분만 고치고 마커를 갱신한다. 비용 ≈ 0.
2. **구조적 갱신 (지문 불일치)**: 훅이 실시간 지문 비교로 달라진 항목 목록
   (매니페스트·CI·빌드 설정 파일명 또는 directory-structure)을 준 경우.
   그 항목들과 관련된 섹션만 검증·수정한다. 아래 절차 중 1단계는 생략하고
   훅이 준 목록에서 시작한다. 지문은 작업 트리의 실제 내용 기준이므로
   미커밋 변경·롤백도 포함되어 있다 — git 히스토리에 없다고 무시하지 마라.
3. **전체 증분 갱신** (20커밋 백스톱): 아래 절차 전체를 수행한다.

## 마커 재기록 (모든 모드 공통 — 빼먹으면 훅이 계속 갱신을 지시한다)

수정을 마치면 문서 상단 마커를 반드시 다음으로 교체한다:

```bash
HEAD=$(git rev-parse HEAD)                # → generated-at-commit 값
FP=$(<hooks>/context-fingerprint.sh)      # → 지문 블록 내용 (그대로 붙여넣기)
```

`<hooks>/`는 배치에 따라 해석한다: `.claude/hooks/`에 스크립트가 있으면
그것을(install.sh 배치), 없으면 이 SKILL.md 기준 `../../hooks/`를(플러그인
배치 — 두 배치 모두 skills/와 hooks/가 같은 루트의 형제라 이 상대 경로는
항상 성립한다) 쓴다.

지문 블록은 직접 계산하지 말고 반드시 생성기 출력을 사용한다.
생성기 출력이 비어 있으면(종료 코드 3 — 해시 도구 없는 환경) 지문 블록을
넣지 마라 — 빈 블록은 거짓 불일치를 만든다. generated-at-commit만 기록한다.

## 출력 언어

`.cursor-context/config`에 `LANG=en`이 설정돼 있으면(기본값은 en이다 —
README가 영어 우선이므로) project-context.md **본문**을 한국어 대신
영어로 작성한다. 이 SKILL.md 자체의 절차 설명은 번역 대상이 아니다 —
읽는 주체가 사람이 아니라 Claude이므로 언어와 무관하게 동일하게 이해한다.
config가 없거나 `LANG=ko`면 한국어로 작성한다.

## 소유권 규칙 (가장 중요)

- **수정 대상은 오직 `.cursor-context/project-context.md`이다.**
- **CLAUDE.md는 사용자가 관리하는 지침 파일이다. 사용자가 명시적으로 요청하지
  않는 한 절대 수정하지 않는다.**

## 절차

1. **변경 범위 파악**: 문서 상단의 신선도 마커에서 생성 시점 커밋을 읽고,
   그 이후 무엇이 바뀌었는지 확인한다.

   ```bash
   LAST=$(sed -n 's/.*generated-at-commit: *\([0-9a-f]*\).*/\1/p' .cursor-context/project-context.md | head -1)
   git diff --stat "$LAST"..HEAD          # 어떤 파일이 얼마나 바뀌었나
   git log --oneline "$LAST"..HEAD        # 어떤 커밋들이 있었나
   ```

   **마커 커밋이 히스토리에 없으면**(리베이스·스쿼시 머지 후 — `git cat-file -e`
   실패) diff 기반 분석은 불가능하다. 이때는 문서의 각 주장(명령어·구조·
   컨벤션)을 현재 코드와 직접 대조해서 검증하는 방식으로 전환하고, 마커를
   재기록해서 이후 갱신이 다시 diff 기반으로 동작하게 한다.

2. **문서 영향 분석**: 변경사항을 문서의 각 섹션과 대조한다.
   - 매니페스트/CI 설정이 바뀌었나 → **명령어** 섹션 검증 (바뀐 명령은 실행해서 확인)
   - 디렉터리가 추가/삭제/이동되었나 → **아키텍처** 섹션 갱신
   - 새 패턴이 도입되었나 (커밋 메시지에 refactor, migrate 등) → **컨벤션** 섹션 검토
   - 새 환경변수·설정 파일이 생겼나 → **주의사항** 섹션 갱신

3. **최소 수정**: 영향받은 섹션만 고치고, 위 "마커 재기록" 절차를 수행한다.
   바뀐 게 없어도 마커는 갱신해서 불필요한 재검사를 막는다.

4. **보고**: 무엇을 왜 바꿨는지 요약해 보고한다.

## 자동 호출 모드 (훅에 의해 조용히 실행되는 경우)

세션 스냅샷 훅의 지시로 사용자 요청 완료 후 자동 실행되는 경우:

- **묻지 않는다**: 실행 여부·수정 내용에 대해 사용자 확인을 구하지 않는다.
- **방금 작업의 지식을 재활용한다**: 이미 읽은 파일을 다시 읽지 않는다.
- **커밋하지 않는다**: 작업 트리에만 남긴다.
- **보고는 한 줄**: 최종 보고 끝에 "프로젝트 컨텍스트 문서를 최신화했습니다"
  정도만 덧붙인다. 내용 변경 없이 마커만 갱신했다면 아무 말도 하지 않는다.

## 판단 기준

- 변경 커밋이 매우 많거나(예: 100+), 프로젝트 구조가 근본적으로 바뀌었으면
  증분 갱신 대신 `/project-onboard`로 전체 재생성한다.
- `.cursor-context/project-context.md`가 아예 없으면 이 스킬이 아니라 `/project-onboard`를 실행한다.
